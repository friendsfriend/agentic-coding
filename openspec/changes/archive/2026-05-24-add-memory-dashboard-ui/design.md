## Context

The current memory extension exposes `/memory show|status|pin|forget|refresh`. The related `add-memory-observability-benchmark` change introduces persistent telemetry, `/memory stats`, `/memory benchmark`, JSON/JSONL benchmark artifacts, and Markdown reports. Those artifacts are useful, but they are still report-oriented: they answer what happened after aggregation rather than giving users an interactive way to explore memory effectiveness.

Pi supports custom TUI components through `ctx.ui.custom()`, overlays, built-in list/detail primitives such as `SelectList`, rendered Markdown, persistent status indicators, and custom components with keyboard input. The dashboard should use these existing primitives and read the same repo-local memory telemetry files instead of introducing a new storage backend.

## Goals / Non-Goals

**Goals:**
- Add `/memory dashboard` as an interactive browser for memory effectiveness.
- Provide overview, benchmark runs, memory entries, and recent turns views.
- Let users drill from summary numbers into the benchmark run, memory entry, or turn that produced them.
- Use compact terminal-friendly visuals such as ASCII bars, deltas, badges, and summary tables.
- Reuse `.pi/memory/stats.jsonl` and `.pi/memory/benchmarks/**` generated by memory observability.
- Keep the dashboard useful even when telemetry is sparse or provider usage fields are unknown.

**Non-Goals:**
- Building a graphical UI outside pi's TUI.
- Replacing Markdown benchmark reports or `/memory stats`; the dashboard complements them.
- Adding cloud telemetry, external charting libraries, or a database.
- Making destructive memory changes the primary v1 dashboard workflow.

## Decisions

### Build a list/detail dashboard first

The dashboard will start as a keyboard-driven list/detail interface with tabs for Overview, Benchmarks, Memories, and Turns. Users navigate with arrow keys, switch views with left/right or tab-like controls, open details with enter, go back or close with escape, and refresh with a keybinding.

Alternative considered: full custom multi-pane dashboard from the start. Deferred because a list/detail UI using existing primitives is faster to implement, easier to test, and less likely to violate terminal width constraints.

### Use generated telemetry and benchmark files as the data source

The dashboard reads aggregated data from `.pi/memory/stats.jsonl`, `.pi/memory/entries.json`, `.pi/memory/benchmarks/**/report.json`, and related Markdown reports where available. It should not introduce separate dashboard-only telemetry files except optional lightweight UI preferences.

Alternative considered: maintaining a separate dashboard cache. Rejected because it could drift from the authoritative local telemetry and benchmark outputs.

### Use pi TUI primitives before custom rendering

Use `ctx.ui.custom()` for the command, `SelectList` for selectable lists, `Markdown` for detailed report rendering, `Text`/`Container` for summaries, and custom render helpers only for compact bars, tables, and badges. All render output must respect the width passed to `render(width)` and use `truncateToWidth` or wrapping utilities for long lines.

Alternative considered: hand-rendering every dashboard view. Rejected because pi's TUI docs recommend reusing existing components for selection and settings-style interactions.

### Keep actions safe and inspectable

The v1 dashboard should support safe actions such as refresh data, open benchmark report, inspect memory entry, inspect turn, and close. Potentially destructive actions such as forgetting a memory entry should require confirmation or remain out of v1 if confirmation would complicate the UI.

Alternative considered: adding one-key forget/pin actions immediately. Deferred or guarded because accidental changes from an observability dashboard would reduce trust.

### Degrade gracefully when data is missing

If telemetry files do not exist yet, the dashboard shows an empty-state explanation and suggests `/memory stats` after observed turns or `/memory benchmark` after implementation. If provider costs are unknown, fields are shown as unknown instead of zero.

Alternative considered: failing with an error when benchmark telemetry is missing. Rejected because the dashboard may be opened before any benchmark has run.

## Risks / Trade-offs

- Dashboard depends on telemetry shape from another active change → Keep parsing tolerant, document expected files, and handle missing fields gracefully.
- Terminal width can break custom tables → Use truncation/wrapping utilities and simple layouts before complex panes.
- Interactive UI can become too large for one extension file → Factor dashboard parsing/rendering helpers into separate modules if implementation grows.
- Users may expect live updates while benchmarks run → Start with manual refresh and progress status; live streaming can be added later if needed.
- Safe actions may feel limited → Prioritize trust and inspection first, then add confirmed actions after the browsing model is stable.

## Migration Plan

1. Add dashboard command routing under the existing `/memory` command surface.
2. Implement tolerant readers and aggregators for stats, memory entries, benchmark summaries, and recent turn telemetry.
3. Build Overview, Benchmarks, Memories, and Turns views using list/detail primitives.
4. Add keyboard navigation, refresh, report opening, and graceful empty states.
5. Validate rendering on narrow and wide terminal widths.
6. Add optional confirmed actions only after basic browsing works.

Rollback is removing the dashboard command and UI helpers. Telemetry and benchmark files remain usable by `/memory stats` and Markdown reports.

## Open Questions

- Should `/memory dashboard` be a full-screen custom component or an overlay side panel?
- Should forgetting or pinning memory entries from the dashboard be in v1, or should v1 remain read-only?
- Should dashboard view preferences, such as last selected tab, persist across sessions?
